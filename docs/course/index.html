What is NebularCore?

You built an app with AI. Let's see what's actually running underneath.

Backend in a Box

NebularCore is a ready-made backend framework — a toolkit that gives your app user login, database storage, file uploads, and more, without building those from scratch.

Go 1.24+ Gin Router GORM + PostgreSQL JWT Auth Event System Multi-Tenancy

Why this matters for you

When you tell AI "add social login" or "require authentication for this route," you need to know WHERE in this chain to make changes — and NebularCore makes each step a distinct, swappable piece.

The toolkit under the hood

NebularCore is itself built from well-known open-source libraries. Think of it as a curated stack of lego pieces, snapped together into a coherent shape.

🌐
Gin

The HTTP router. Every request your app receives passes through Gin first.

🗄️

The ORM. Translates Go code into database operations so you can say 'save this user' instead of writing SQL.

📨

The event bus. Lets modules talk to each other without being tangled together.

🔒

The authorization engine. Defines rules like "admins can delete, editors can only edit."

🐍

The CLI framework. Gives you commands like nebularcore serve and nebularcore migrate up.

Your first 10 lines of NebularCore

Here's what an app that uses NebularCore actually looks like at startup. This is from the examples/main.go file — the front door of the system.

CODE — examples/main.go 
func main() {
  app := nebularcore.New(core.Options[ExampleSettings]{
    ConfigPath: "./config.yml",
    EnvPrefix:  "NEBULAR",
  })

  eventBus, _ := event.New()
  app.RegisterModule(eventBus)

  authModule := auth.New(eventBus)
  app.RegisterModule(authModule)

  app.Bootstrap(context.Background())
  app.Start(ctx)
}
PLAIN ENGLISH

Create the app — read the config file and set up everything from it.

ConfigPath — the YAML file with database credentials, server port, etc.

EnvPrefix — environment variables starting with NEBULAR_ can override config values.

Create the event bus — a messaging channel all modules can share.

RegisterModule — plug the event system into the app (like installing a plugin).

Create auth module — it needs the event bus so it can announce "user logged in."

Bootstrap — run all the setup code: connect to database, initialize each module in the right order.

Start — begin listening for HTTP requests. The app is now live.

The Module System

How NebularCore turns a chaotic pile of features into an orderly, dependency-aware system.

Every module signs a contract

Think of NebularCore as a professional kitchen. To work there, every chef must follow the same protocol: show up, prep your station, cook when asked, clean up when done. In code, this protocol is called an interface.

CODE — core/module/module.go 
type Module interface {
  Name()    string
  Version() string
  Dependencies() []string
  Namespace() ModuleNamespace

  Initialize(ctx context.Context,
    db *gorm.DB,
    router *gin.Engine) error
  NewConfig() any
  Configure(config any) error
  Shutdown(ctx context.Context) error

  ProvidesMigrations() bool
  GetMigrationSources(...) []Source
}
PLAIN ENGLISH

Name() — What are you called? (e.g. "auth", "storage")

Version() — Which version are you? Useful for debugging.

Dependencies() — Who do you need to be initialized before you? Return a list of module names.

Namespace() — Do you operate in the shared public schema, or in each tenant's private schema?

Initialize() — Run your setup code here. You get the database, the HTTP router, and a context. Register your routes, set up your tables.

NewConfig() / Configure() — Describe what config values you need; the framework will load them from the YAML file and hand them to you.

Shutdown() — Clean up when the server stops: close connections, flush buffers, say goodbye.

ProvidesMigrations() — Do you have SQL files that need to run to set up your database tables? Tell the framework here.

Public vs Tenant namespace

This is the concept that makes multi-tenancy work. Imagine a shared office building with locked private suites. The database schema is like office space — there's the shared lobby (public schema) and private offices for each tenant.

🌐
PublicNamespace

Modules like auth, health, and storage operate here. Their database tables live in the shared public schema. A user account is shared across all tenants.

🏢
TenantNamespace

Modules in TenantNamespace get their tables in each tenant's private schema. A record created by Tenant A is completely invisible to Tenant B — guaranteed by the database.

💡
When to use this in practice

If you're building a SaaS where each customer's data must be isolated (like their own CRM records), tell your AI agent: "This module should use TenantNamespace so each customer's data is schema-isolated." That's the exact vocabulary NebularCore uses.

Auth & Security

How NebularCore proves who you are, keeps passwords safe, and issues digital VIP passes.

Security like a venue with wristbands

Picture a music festival. At the gate, security checks your ID (authentication). If valid, they give you a wristband (JWT token). After that, every time you try to enter a stage or backstage, staff check the wristband color — not your ID again (authorization). The wristband is self-contained proof.

🪪
Authentication

"Who are you?" — checked at login using email+password, Google OAuth, or Apple Sign-In. Handled by the Auth Module's backend system.

🎟️
JWT Token

"Here's your wristband." A signed string returned after login. Contains the user's ID and expiry. Your app sends it in every subsequent request inside the Authorization header.

🛡️
Middleware

"Check the wristband." A middleware function that intercepts every protected request, validates the JWT, and attaches the user's identity to the request context.

Step 0 / 6

The auth handler, decoded

Here's the real code that registers the auth routes in NebularCore — straight from modules/auth/handlers/auth.go:

CODE — modules/auth/handlers/auth.go 
func (h *AuthHandler) RegisterRoutes(
  router *gin.RouterGroup,
  socialSignupHandler gin.HandlerFunc,
) {
  auth := router.Group(h.basePath)
  {
    if h.config.Social.Enabled {
      auth.GET("/providers/:provider/initiate",
        h.InitiateSocialLogin)
      h.RegisterSocialSignupRoutes(
        router, socialSignupHandler)
    }
    auth.POST("/login",      h.Login)
    auth.POST("/refresh",    h.RefreshToken)
    auth.POST("/logout",     h.Logout)
  }
}
PLAIN ENGLISH

RegisterRoutes — "set up all my URLs." Called during Bootstrap.

auth.Group(h.basePath) — All routes will be prefixed with /auth (so /auth/login, /auth/logout, etc.)

if h.config.Social.Enabled — Only register Google/Apple login routes if social auth is turned on in the config file.

/providers/:provider/initiate — The :provider part is a URL variable — user can visit /auth/providers/google/initiate to start Google login.

POST /login — The email+password login endpoint. Calls h.Login which checks credentials and issues a JWT.

POST /refresh — When the JWT is about to expire, the client calls this to get a fresh one without logging in again.

POST /logout — Invalidates the current session.

How passwords are stored safely

NebularCore never stores your password in plain text. Instead, it uses bcrypt — a one-way hashing algorithm. Think of it as a shredder: once your password goes in, only confetti comes out. To check a password, you shred it again and compare the confetti patterns.

🔑
What this means for you

"Can you store this password?" → Yes, using bcrypt. "Can you look up a user's password?" → No one can, not even an admin. "Can you reset it?" → Yes, by generating a new one. This is the gold standard and NebularCore implements it by default.

🌐
Social login (Google, Apple, Facebook)

With social login, the user never gives NebularCore a password. Instead, Google/Apple verifies the identity and hands NebularCore a signed certificate. NebularCore checks the certificate is genuine, then issues its own JWT. No password stored anywhere.

Events & Migrations

How NebularCore keeps modules talking without tangling, and how it evolves your database safely over time.

The event bus: a postal service for code

Imagine every building in a city has a mailbox. When something happens — a user signs up, a payment succeeds — a letter is dropped into the central post office. Any building that cares about that event has signed up to receive that type of letter. Nobody calls each other on the phone — they just check the mail.

NebularCore's event system (Watermill) works exactly like this. Here's the code that publishes a login event:

CODE — modules/event/event.go 
func (m *Module) Publish(
  ctx context.Context,
  event Message) error {

  event.Metadata.Set("source", event.Source)
  event.Metadata.Set("event_type", event.EventType)
  event.Metadata.Set("timestamp",
    event.Timestamp.Format(time.RFC3339))

  return m.publisher.Publish(
    event.EventType, event.Message)
}
PLAIN ENGLISH

Publish — "drop a letter in the post office." Any module can call this.

Metadata.Set("source") — Stamp the envelope with who sent it (e.g., "auth module").

Metadata.Set("event_type") — What kind of letter is this? (e.g., "user.logged_in"). Subscribers filter by this.

Metadata.Set("timestamp") — When exactly did this happen? Useful for debugging and audit trails.

m.publisher.Publish(event.EventType, event.Message) — Actually drop the letter. Everyone subscribed to this event_type receives it.

Sync vs. Async publishing

NebularCore offers two ways to publish events. The difference matters when you need speed vs. reliability:

⏳
Publish() — Synchronous

Waits for the event to be received before continuing. If delivery fails, it returns an error right away. Use when you need to know the event was delivered before proceeding.

⚡
PublishAsync() — Async

Fires the event in the background and returns immediately. Your code keeps running at full speed. Errors come back on a channel you can check later. Use for non-critical notifications (analytics, emails).

💡
When to tell your AI agent which to use

"Send the welcome email asynchronously after registration" → PublishAsync(). "Don't continue until the payment event is confirmed" → Publish(). Choosing the right one prevents subtle bugs where an app appears fast but silently drops important actions.

Migrations: how your database evolves safely

Imagine your database is a city. When the city was first built, there were 10 streets. As it grew, you added new streets and renamed old ones. A migration is the city planner's official record of every change, in order, so any new installation can replay the full history and end up with the exact same city.

1

Each module has its own migrations/ folder with numbered SQL files

→
2

The Migration Runner reads all sources, sorted by priority

→
3

It checks which migrations have already run (tracked per module)

→
4

Applies only the new ones, in order, atomically

📝
The .up.sql / .down.sql pattern

Every migration has two files: 000001_create_users.up.sql (apply the change) and 000001_create_users.down.sql (reverse the change). If something goes wrong in production, you can roll back to exactly where you were. NebularCore even lets you override framework migrations with your own project-level versions — using the priority + exclude system.

Three patterns that make NebularCore reusable

These aren't NebularCore-specific ideas — they're universal software engineering patterns you'll see in every well-built backend. Understanding them makes you a better director of AI tools.

🧩
Interface-Based Design

Everything in NebularCore works against interfaces, not concrete types. Your module doesn't care how storage works — just that it has Upload() and Download(). Swap S3 for GCS and nothing else changes. This is the dependency inversion principle.

🏭
Factory Pattern

The auth module uses a UserFactory interface — you provide a factory that creates your custom User struct, and the auth module uses it without knowing your struct's fields. This is how NebularCore stays generic while you stay specific.

📦
Generic Types

NebularCore uses Go generics: App[T config.Settings] means your app carries your exact settings type with full type safety — no casting or type-checking at runtime.

The Live Wire

Real-time bidirectional communication — WebSockets explained without the jargon.

HTTP vs WebSockets: letter vs phone call

HTTP is like sending a letter. You post it, wait for one reply, then post again. Every message requires a brand-new round trip. WebSockets are like a phone call — the line stays open and both sides can speak any time without redialling.

NebularCore's WebSocket module adds that always-open phone line to your backend with a single GET /ws endpoint.

🔌
One endpoint, many connections

A single GET /ws route handles every real-time client. Gin upgrades HTTP to WebSocket (HTTP 101). From that moment, data flows freely both ways.

🗺️
Topic subscriptions

Clients subscribe to named topics (e.g. orders.placed). When that event fires anywhere in the app, the EventBridge fans it out only to subscribed connections — no polling needed.

🛡️
Same JWT, no new auth

The WebSocket middleware reads the same JWT your REST API issues. One token, one identity — WebSockets are just another surface, not a separate auth system.

⚡
Sharded connection manager

Connections are stored across 256 independent shards. No single lock blocks the whole map — thousands of concurrent connections stay fast.

Reading Initialize() — the wiring code

When NebularCore boots, the WebSocket module's Initialize() method runs. Here's what each line actually does:

CODE — modules/websocket/module.go 
func (m *Module) Initialize(
  ctx context.Context,
  db  *gorm.DB,
  ginRouter *gin.Engine,
) error {
  m.manager = connections.NewManager(
    m.config.Server.MaxConnections)

  m.pool = connections.NewPool(
    m.manager,
    m.config.Server.MaxConnectionsPerUser,
    m.config.Server.MaxConnectionsPerTenant)

  m.subs   = store.NewSubscriptions()
  m.router = bridge.NewRouter()

  handlers.RegisterMessageHandlers(
    m.router, m.manager,
    m.subs, m.eventBus,
    m.config.Routing.MaxTopicsPerConnection)

  if len(m.config.Events.AllowedEventTypes) > 0 {
    m.evBridge = bridge.NewEventBridge(
      m.eventBus, m.manager,
      m.subs,
      m.config.Events.AllowedEventTypes)
    m.evBridge.Start(ctx)
  }

  ginRouter.GET("/ws",
    wsauth.Middleware(jwtSecret, authRequired),
    wsHandler.Handle)
}
PLAIN ENGLISH

NewManager — Creates the sharded connection store. Think of it as a massive switchboard that can handle thousands of open lines at once without slowing down.

NewPool — Adds rate-limit guards: "no more than N connections per user, and no more than M per tenant." Prevents one bad actor from hogging all the lines.

NewSubscriptions — Creates the topic registry: a map of "who is listening to what." When a client says subscribe orders.placed, it lands here.

RegisterMessageHandlers — Wires up handlers for each JSON message type the client can send: subscribe, unsubscribe, publish, ping.

EventBridge.Start — Plugs into the internal event bus. Every time an allowed event fires, it fans out to all WS clients subscribed to that topic.

GET /ws — Registers the upgrade endpoint on Gin, protected by JWT middleware — same token, no extra login needed.

The message protocol & live simulator

Every message between client and server is plain JSON with a type field. The router reads that field and dispatches to the right handler — like a tiny switch statement per connection.

→
Client → Server

publish — Broadcast to all topic subscribers
ping — Keep-alive heartbeat

←
Server → Client
